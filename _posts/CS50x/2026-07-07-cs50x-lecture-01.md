## 1. 컴퓨터 과학

- 컴퓨터는 전기가 **"흐른다(1)" / "흐르지 않는다(0)"** 라는 두 가지 상태만 명확하게 구분하면 됨
- 이 비트(0 또는 1)는 실제로 **트랜지스터의 켜짐/꺼짐**, 즉 고전압/저전압 상태로 구현됨
- 두 가지 상태만 쓰는 이유는 **십진법(0~9, 10가지 전압 수준을 구분)보다 하드웨어 구현이 훨씬 단순**하고, 노이즈에도 강하기 때문
- 그래서 손가락 실습에서 배운 "자리마다 가중치를 부여하는" 이진법 원리가, 실제 컴퓨터 내부의 트랜지스터(켜짐/꺼짐)로 그대로 이어짐
- 즉 "든 손가락(1)/접은 손가락(0)"이 컴퓨터에서는 **"전류 흐름(1)/안 흐름(0)"** 으로 정확히 대응되는 구조

## 2. ASCII

- 과거 사람들이 알파벳과 숫자를 하나씩 대응시켜 표로 정리해둔 표준 규약으로 대문자 A = 65부터 시작해서 B=66, C=67 순서로 이어진다.
- 처음엔 7비트로 시작했지만 이후 8비트(1바이트)로 표현하는 것이 일반화되었다.

- 소문자는 대문자보다 정확히 **32만큼 큰 값**을 가진다.
- 즉 컴퓨터가 대문자를 소문자로 바꾸려면, 32를 더하는 것과 같은 효과를 내는 비트 하나만 바꾸면 된다 (32는 2⁵이므로 특정 한 자리 비트를 1로 바꾸는 것과 동일)
- 이와 같이 이진수의 자릿값 개념이 실제로 문자 처리에도 그대로 활용된다

[ASCII의 한계]

- 7~8비트로는 최대 256가지 패턴만 표현 가능하다.
- 영어 알파벳(대소문자), 숫자, 기본 문장부호 정도는 충분히 담을 수 있지만, 억양이 있는 문자나 아시아권 문자, 다양한 특수기호는 담을 공간이 부족하다.
- 이 한계 때문에 이후 등장하는 것이 유니코드(Unicode)이다.

![ASCII 코드표](/assets/img/cs50x/cs50x-01-ascii-table.png){: .normal width="600"}

## 3. Unicode

- 유니코드 컨소시엄이라는 단체가 과거·현재·미래의 모든 인간 언어를 디지털로 표현하려는 목표로 만든 확장된 표준이다.
- 8비트가 아니라 16비트, 24비트, 심지어 32비트까지 사용해 훨씬 많은 문자를 표현한다.
- **ASCII**를 사용할 경우 8비트로 실제로 표현 가능한 패턴 수가 256가지인 반면,
  **기본 유니코드**는 16비트로 65,536가지 패턴을 표현 가능하며, **확장된 유니코드**는 32비트로 확장되어 약 40억가지 패턴을 표현 가능하다.

- 이렇게 많은 패턴이 확보된 덕분에 각종 이모지까지 문자 체계 안에 포함시킬 수 있게 되었다.
  - 이모지는 이미지가 아니라 문자이며, 유니코드는 "의미"만 정의하고 "모양"은 각 회사가 자유롭게 디자인한다.
  - 이 원리는 앞서 배운 "**같은 비트 패턴도 문맥에 따라 다르게 해석될 수 있다**"는 개념과 연결된다.

![유니코드 한글 인코딩 예시](/assets/img/cs50x/cs50x-01-unicode-hangul.png){: .normal width="600"}

## 4. Color RGB

- 문자를 표현할 때처럼, 색도 "어떤 숫자를 어떤 색으로 볼지" 약속이 있으며
  가장 널리 쓰이는 방식은 **빨강(Red), 초록(Green), 파랑(Blue)** 세 가지 색을 섞어서 원하는 색을 만드는 RGB 방식이다.
  - 옛날 영사기가 세 개의 렌즈(빨강·초록·파랑)로 빛을 겹쳐서 화면에 다양한 색을 만들어내던 원리와 동일하다.

[RGB 값의 구조]

- 빨강, 초록, 파랑 각각 1바이트(8비트)씩 사용 → 값의 범위는 0~255
- 0은 "그 색이 전혀 없음", 255는 "그 색이 최대치로 있음"을 의미

- R=72, G=23, B=33이라는 숫자 조합이, ASCII 관점에서 보면 각각 H, I, ! 라는 문자와 같은 숫자이다.
- 즉 **똑같은 비트 패턴이라도, 프로그램이 "이걸 문자로 봐라" 하면 문자가 되고, "이걸 색으로 봐라" 하면 색이 된다**는 원칙을 확인할 수 있다.
- 웹페이지나 포토샵에서 자주 보는 `#00` 또는 `#FF` 같은 표기는, 0~255를 표현하는 또 다른 진법(16진법)일 뿐, 근본 원리는 동일하다.

## 5. Pixels, Images, Video, and Music

### 5-1 Pixels, Images

- 이미지를 확대하면 결국 작은 점들의 격자로 이루어져 있음을 확인할 수 있다. 작은 점 하나하나가 픽셀이다.
- 각 픽셀은 RGB 방식이라면 R, G, B 각각 8비트씩, 총 24비트(3바이트)의 색상 정보를 가진다.
- 즉, 이미지 전체는 수많은 픽셀 각각의 색상 정보 모음이다.
- 이미지 하나에 픽셀이 수천~수백만 개 있고, 픽셀 하나당 최소 3바이트가 필요하므로 금방 수백 KB~수 MB 단위로 커진다.
- 다운로드한 사진이나 GIF 파일의 용량이 보통 메가바이트 단위인 이유가 바로 이것이다.

### 5-2 Video

- 영상은 결국 정지된 이미지 여러 장이 초당 약 30장 정도의 속도로 빠르게 넘어가는 것이다.
- 사람 눈이 이를 부드러운 움직임으로 착각하는 것이며 어릴 때 갖고 놀던 플립북(빠르게 넘기면 그림이 움직이는 것처럼 보이는 책)과 동일한 원리이다.
- "motion pictures(영화)"라는 옛 표현도 이 원리에서 유래되었다.

### 5-3 Music

- 음표 하나를 표현하려면 최소 세 가지 정보가 필요하다. 음높이(주파수), 길이(지속 시간), 음량(세기)
- 각각을 숫자(예: 0~255 범위)로 표현하면, 이 세 숫자 묶음으로 음표 하나를 나타낼 수 있다.
- 받는 쪽이 이 숫자들을 같은 방식으로 해석할 줄 알면, 작곡자가 의도한 그대로의 소리를 재현할 수 있게 된다.

[결론]

- 이미지, 영상, 음악 모두 결국 0과 1의 나열일 뿐이다.
- 프로그램(소프트웨어)이 특정 패턴을 색으로 보는 것인지, 음표로 보는 것인지 정확히 해석해줄 때만 우리가 의도한 대로 보이고 들린다.
- 이는 앞서 문자·색상에서 확인한 원칙이 미디어 전반에 일관되게 적용됨을 보여준다.

## 6. Algorithm

- 수백~수천 페이지 분량의 전화번호부(또는 연락처 앱)에서
  특정 사람의 이름을 찾아야 할 때 같은 문제를 푸는 방법(알고리즘)은 여러 가지가 있을 수 있고, 그 방법들의 "효율성"이 다르다

![전화번호부 탐색 방법 비교](/assets/img/cs50x/cs50x-01-search-methods.png){: .normal width="400"}

![Big-O 그래프: O(n)과 O(log n)](/assets/img/cs50x/cs50x-01-big-o-graph.png){: .normal width="600"}

- 전화번호부 페이지가 1000페이지에서 2000페이지로 두 배가 되어도, 이진 탐색은 겨우 한 단계만 더 나누면 되는 반면 한 장씩 넘기는 방법은 시간이 그대로 두 배가 된다.
- 그래프로 그리면 O(n)은 직선으로 계속 증가하지만, O(log n)은 아주 완만하게 휘어지는 곡선(로그 그래프) 형태가 된다.
- 좋은 컴퓨터과학은 **단순히 정답을 맞히는 것이 아니라, 제한된 자원(CPU, 메모리, 시간, 인력, 비용)을 최소로 쓰면서 문제를 해결하는 것을 목표**로 한다.
- 이 예시는 실제로 스마트폰 연락처 앱의 자동완성 기능이 왜 그렇게 빠른지 설명하는 배경이 되며 목록이 알파벳순으로 정렬되어 있어야 이진 탐색 같은 효율적인 방법을 쓸 수 있다.

## 7. 의사코드와 핵심 개념

- 이진 탐색으로 전화번호부에서 이름을 찾는 과정을 실제 코드로 옮기기 전, 사람이 이해하기 쉬운 "의사코드(pseudocode)"로 먼저 표현
- 이 과정에서 함수, 조건문, 불리언 표현식, 반복문이라는 프로그래밍의 핵심 개념이 자연스럽게 등장한다.

### 7-1. 의사코드

- 정해진 문법이 있는 것이 아니라, 사람마다 자기 나름의 방식으로 표현하는 "인간이 이해할 수 있는 형태의 단계별 지시문"
- 실제 프로그래밍 언어 문법을 몰라도, 문제 해결 절차를 명확한 언어로 적어보는 연습이 필요하다.
- 아래 표의 네 가지 개념은 Scratch, C, Python 등 어떤 언어를 쓰든 코드의 뼈대를 이루는 공통 요소이다.
- 예외 상황(코너 케이스)을 놓치지 않고 처리하는 것이 정확하고 안정적인 프로그램을 만드는 핵심이다.

| 개념 | 의미 |
|---|---|
| 함수(Function) | 어떤 작업을 실제로 수행하는 동작/명령 (동사에 해당) |
| 조건문(Conditional) | 갈림길 — 상황에 따라 다른 경로로 분기 |
| 불리언 표현식(Boolean Expression) | 참/거짓, 예/아니오로만 답할 수 있는 질문 |
| 반복문(Loop) | 같은 동작을 조건이 만족되는 동안 반복 수행 |

## 8. Abstraction(추상화)

- 알고리즘을 실제 코드로 옮기는 과정에서, 컴퓨터가 이해하는 0과 1부터 사람이 다루기 쉬운 Python까지 어떻게 계층이 쌓여왔는지 설명한다.
- 또한 "추상화(abstraction)"라는 컴퓨터과학의 핵심 원리를 소개한다.

- 명령어도 결국 0과 1의 패턴이다.
  - 데이터(숫자, 문자, 색상)뿐 아니라 "무엇을 하라"는 명령어도 결국 특정 비트 패턴으로 표준화되어 있음
  - 예를 들어 특정 패턴은 "두 수를 더하라", 다른 패턴은 "화면에 출력하라"는 의미로 회사(인텔, AMD, 엔비디아 등)마다 정해둠
  - 즉 "Hello World"를 출력하는 프로그램도, 맨 밑바닥에서는 결국 0과 1로 된 명령어 나열일 뿐

### 8-1 계층구조

- 계층구조 : 기계어 → C → Python

| 계층 | 특징 |
|---|---|
| 기계어(0과 1) | 컴퓨터가 직접 이해하는 가장 낮은 수준. 사람이 다루기엔 매우 번거로움 |
| 어셈블리/펀치카드 시대 | 초기에는 사람이 직접 0과 1에 가까운 형태로 명령을 작성해야 했음 |
| C 언어 | 사람이 좀 더 읽을 수 있는 문법으로 명령을 작성 → 컴파일러가 기계어로 번역 |
| Python | C보다 훨씬 사람 친화적인 문법 → 내부적으로는 결국 C 등 하위 계층을 거쳐 기계어로 변환됨 |

[컴파일러]

- 한 언어로 작성된 코드를 다른 언어(궁극적으로는 기계어)로 번역해주는 프로그램이다.
- 누군가 "0과 1을 직접 다루는 게 너무 번거롭다"고 느껴서, 좀 더 사람 친화적인 언어(C)를 만들고 그것을 기계어로 자동 번역해주는 도구를 만든 것을 시작으로
  이후 C보다 더 간단한 Python이 등장했고, Python 코드도 결국 C 등을 거쳐 기계어로 변환되는 과정을 거친다.

### 8-2 추상화

- 낮은 수준의 복잡한 구현 세부사항을 몰라도, 그 위에 쌓인 도구를 이용해 문제를 해결할 수 있게 해주는 원리다.
- 우리는 트랜지스터가 어떻게 동작하는지 몰라도 C를 쓸 수 있고, C의 세부 구현을 몰라도 Python을 쓸 수 있으며, Python의 내부를 몰라도 OpenAI API를 이용한 챗봇을 만들 수 있다.
- **"이미 누군가 어려운 문제를 해결해뒀다"는 것을 믿고, 그 위에서 더 높은 수준의 문제를 해결**하는 것이며 이것이 **현대 소프트웨어 개발과 AI 활용의 핵심 원리**
  - 일상적 의미의 추상은 '듣는 사람이 세부사항 없이도 큰 그림을 이해함'
  - 개발적 의미는 개발자가 낮은 수준의 구현 디테일을 걷어내고 "기능"만 남김

## 9. 코드의 개념

- 복잡해 보이는 게임도 결국 스프라이트, 코스튬, 좌표, 반복문, 조건문, 변수, 커스텀 함수라는 익숙한 building block들의 조합이다.
  - 게임 안에서 서로 다른 행동을 하는 캐릭터들도 결국 "무한 반복 + 조건문 + 이동/방향 전환"이라는 같은 기본 패턴의 변형일 뿐이다.
- 중복되는 코드는 커스텀 함수로 묶어 유지보수성을 높이는 것이 좋은 설계 방식이다.
- 큰 프로젝트는 작은 버전(Oscar 0 → 1 → 2 → 3 → 4)으로 나누어 단계적으로 발전시키는 것이 효과적인 개발 전략
