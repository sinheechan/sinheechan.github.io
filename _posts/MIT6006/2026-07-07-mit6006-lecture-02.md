## 0. 핵심 질문

- 읽기도 빠르고 추가·삭제도 빠른 자료구조가 있을까? → 완벽한 건 없다. 트레이드오프를 이해하는 게 목표.

## 1. 인터페이스 vs 자료구조

- 인터페이스: 무엇을(What) 할 수 있는가 : 연산의 목록(명세)
- 자료구조: 어떻게(How) 하는가 : 실제 구현 방식 (저장 방식 + 알고리즘)
- 대표 인터페이스 2개: Sequence, Set

## 2. 대표 인터페이스

- Sequence : "순서(위치)"가 핵심. 순서는 외재적(우리가 부여).
  - Container(`build`, `len`)
  - Static(`get_at`, `set_at`)
  - Dynamic(`insert_at`, `delete_at`, `insert/delete_first/last`)
- Set : "키(key)"가 핵심. 순서가 있다면 내재적(키 값에서 나옴).
  - Static(`find`)
  - Dynamic(`insert`, `delete`)
  - Order(`find_min/max`, `find_next/prev`)

## 3. 대표 자료구조

| 자료구조 | 강점 | 약점 |
|---|---|---|
| 정적 배열 | 읽기 `get_at`/`set_at` Θ(1) | 삽입·삭제 시 복사/밀기 Θ(n) |
| 연결 리스트 | 앞쪽 삽입·삭제 Θ(1) (포인터 재연결) | 읽기 `get_at` Θ(n) (순차 탐색) |
| 동적 배열 | 읽기 Θ(1) + 뒤쪽 삽입 Θ(1) 상환 | 앞쪽·중간 조작은 여전히 Θ(n) |

- 동적 배열 아이디어: 여유 공간을 미리 잡아둠 → 꽉 차면(r=1) 두 배로 확장 → 한 번 키우면 Θ(n)번 복사 없이 삽입 가능. (Python `list`)

## 4. 상환 분석 (Amortized)

- 비싼 확장(Θ(n))은 아주 가끔 일어남 → 값싼 연산들에 비용을 분산하면 평균은 저렴.
- 정의: k번 총비용 ≤ `k·T(n)` 이면 상환 비용 `T(n)`.
- 동적 배열 삽입 = Θ(1) 상환.

## 5. 동적 배열의 삭제 (공간 관리)

- 뒤에서 삭제 자체는 Θ(1). 단, 큰 배열을 그대로 두면 공간 낭비.
- 해법: 비율 r이 `r_d`(예: 1/4) 아래로 떨어지면 `r_i`(예: 1/2)로 축소.
- 추가 공간을 `(1+ε)n` 수준으로 제한 가능.

## 6. 최종 요약표

| 연산 | 정적 배열 | 연결 리스트 | 동적 배열 |
|---|---|---|---|
| `build(X)` | n | n | n |
| `get_at`/`set_at` | 1 | n | 1 |
| `insert/delete_first` (앞) | n | 1 | n |
| `insert/delete_last` (뒤) | n | n | 1 (상환) |
| `insert/delete_at` (중간) | n | n | n |
